## Plan: Integration with Other Agent Harnesses

### Current State

Agent Kanban's interaction layer is currently VS Code–specific:

- The **Kanban board UI** (drag-and-drop, task creation) lives entirely in a VS Code WebviewView — it's a pure VS Code extension feature.
- The **`@kanban` chat participant** routes commands (`/new`, `/task`) using VS Code's `ChatParticipant` API — this only works in GitHub Copilot Chat inside VS Code.

However, the **data layer is already fully agent-agnostic**: every task is a plain markdown file with YAML frontmatter in `.agentkanban/tasks/`. Any agent that can read and write files can already work with the task data. The only friction is that external agents don't have a convenient way to:
1. Discover which task to work on
2. Know the workflow conventions (plan → todo → implement)
3. Interact with the board metadata (lanes)

---

### Integration Approaches by Harness Type

#### CLI-based agents (Codex CLI, OpenCode, Aider, etc.)
These agents run from a terminal, typically in the project root, and read their own instruction files (e.g. `AGENTS.md`, `.aider.conf.yml`). They can access the filesystem freely.

**How they can work today (no code changes):**  
A developer can manually open a task file and paste its path as context. The agent can then read and write to it following the INSTRUCTION.md conventions.

**Gap:** No automated task selection or board state awareness.

#### Claude Code (VS Code extension + `claude` CLI)
Reads `CLAUDE.md` at the project root. The CLI variant (`claude`) operates outside VS Code entirely.

**How they can work today:**  
A developer can add a reference to `.agentkanban/INSTRUCTION.md` inside `CLAUDE.md`. Claude Code will then follow the same workflow conventions. Task selection is still manual.

#### Cursor / Continue.dev
Both are IDE-based and support rule/instruction files. They don't support VS Code Chat Participants.

**How they can work today:**  
Via instruction files pointing at `.agentkanban/INSTRUCTION.md`. Board interaction is limited to editing markdown files directly.

---

### Proposed Integration Strategies (ordered by value/effort)

#### Strategy 1: MCP Server (highest value, medium effort)
An **MCP (Model Context Protocol) server** is the most universal solution. MCP is supported by Claude Code, Cursor, GitHub Copilot, Windsurf, and others. An MCP server could expose:

| Tool | Description |
|------|-------------|
| `listTasks` | Return all tasks with id, title, lane |
| `createTask` | Create a new task, return file path |
| `getTask` | Return full task content (frontmatter + conversation) |
| `selectTask` | Return task context + workflow instructions for any agent |
| `appendToTask` | Append an agent or user turn to a task file |
| `moveTask` | Change a task's lane |

This approach decouples Agent Kanban from VS Code entirely. The board UI (VS Code extension) becomes a visual layer on top of the MCP server, and any MCP-compatible agent gets full programmatic access.

**Implementation:** A standalone `agentkanban-mcp` package or a built-in MCP server mode in the extension binary.

#### Strategy 2: Lightweight CLI (`agentkanban` npm package, low-medium effort)
A small CLI tool published to npm:

```bash
npx agentkanban list                  # List tasks with lanes
npx agentkanban select "My Task"      # Print task context (for piping into agent prompts)
npx agentkanban create "New Feature"  # Create task, print file path
npx agentkanban move <id> done        # Move task to a lane
```

CLI agents could invoke these commands as shell tools. This is simpler than MCP but works with any harness that can run shell commands (Aider, Codex CLI, etc).

#### Strategy 3: Universal instruction injection (low effort)
Document a standard pattern in README for each major harness: how to reference `.agentkanban/INSTRUCTION.md` from `CLAUDE.md`, `AGENTS.md`, cursor rules, etc. The Kanban board UI (VS Code) still handles task creation/organisation; external agents follow the file-based protocol.

No code changes required. Adds friction (manual task selection) but works today.

#### Strategy 4: Agent-specific companion files (low effort)
On extension activation or on demand, auto-generate harness-specific stubs:
- `CLAUDE.md` snippet referencing INSTRUCTION.md
- An `AGENTS.md` snippet
- A `.cursor/rules` entry

These could be generated by a new command: **Agent Kanban: Generate Agent Instructions**.

---

### Recommended Phased Approach

| Phase | Work | Value |
|-------|------|-------|
| 1 | Strategy 3 — Document universal instruction patterns in README | Immediate, no code |
| 2 | Strategy 4 — `Generate Agent Instructions` command producing harness stubs | Low effort, high discoverability |
| 3 | Strategy 2 — `agentkanban` CLI npm package | Medium effort, broad CLI compatibility |
| 4 | Strategy 1 — MCP server | Highest value, works everywhere MCP is supported |

Phase 4 (MCP) is the long-term north star: it makes Agent Kanban a first-class citizen across the entire modern agent ecosystem, not just VS Code + Copilot.

---

### Key Design Principle to Preserve
The task file format (plain markdown + YAML frontmatter) is the right abstraction. It's git-friendly, human-readable, and works with any agent that can do file I/O. The goal of any integration work is to make the board metadata (lanes, selection) equally accessible without requiring VS Code or Copilot.

## MCP Server: User Experience & Registration

### What is an MCP Server?

An MCP (Model Context Protocol) server is a small process that exposes a set of **tools** to an AI agent host over a standard JSON-RPC 2.0 transport (typically stdio or HTTP/SSE). The agent host calls the tools on behalf of the LLM when it decides to invoke them. The user doesn't directly communicate with the server — it's invisible infrastructure.

---

### User Communication Flow (typical workflow)

From a user's perspective, the experience looks like this:

#### In Claude Code (VS Code extension or CLI)
```
User: "Start working on the OAuth task"

Claude: [calls listTasks → sees "Implement OAuth2" in the doing lane]
        [calls selectTask("task_20260308_...oauth2") → gets task file content + instructions]
        [reads the conversation history from the task file]

Claude: "I see the OAuth2 task. The plan covers 3 areas: token refresh, 
         scope handling, and error states. Shall I start implementing?"

User: "Yes, focus on token refresh first"

Claude: [does implementation work...]
        [calls appendToTask(id, "agent", "Implemented token refresh logic in...")]

Claude: "Done. I've updated the task log. Moving to scope handling..."
```

The user just talks naturally. The MCP tools handle task state management invisibly.

#### In GitHub Copilot (VS Code, via MCP)
Same experience as today's `@kanban` chat participant — but the board operates as MCP tools rather than a VS Code-specific participant. This means the same server works whether the user is in VS Code, the GitHub CLI, or a future Copilot surface.

---

### MCP Server Registration: Per-Harness Setup

Each harness has its own mechanism. Here's what the one-time setup looks like:

#### Claude Code (`~/.claude/settings.json` or local `.claude/settings.json`)
```json
{
  "mcpServers": {
    "agentkanban": {
      "command": "npx",
      "args": ["agentkanban-mcp"],
      "cwd": "${workspaceFolder}"
    }
  }
}
```
Or for a local install:
```json
{
  "mcpServers": {
    "agentkanban": {
      "command": "node",
      "args": [".agentkanban/mcp-server/index.js"]
    }
  }
}
```
Claude Code reads this on startup and spawns the server process. The user sees the tools listed in Claude's tool palette. No further setup required.

#### GitHub Copilot (VS Code, MCP extension support)
In `.vscode/mcp.json` (VS Code workspace):
```json
{
  "servers": {
    "agentkanban": {
      "type": "stdio",
      "command": "npx",
      "args": ["agentkanban-mcp"],
      "env": {}
    }
  }
}
```
VS Code's MCP support (available in recent VS Code + Copilot builds) reads this and makes the tools available in agent mode.

#### Cursor (`.cursor/mcp.json`)
```json
{
  "mcpServers": {
    "agentkanban": {
      "command": "npx",
      "args": ["agentkanban-mcp"]
    }
  }
}
```

#### Windsurf / Codeium (`~/.codeium/windsurf/mcp_config.json`)
```json
{
  "mcpServers": {
    "agentkanban": {
      "command": "npx",
      "args": ["agentkanban-mcp"]
    }
  }
}
```

**The pattern is identical across all harnesses** — `command` + `args` to launch the server process over stdio. The MCP spec mandates this, which is what makes it so portable.

---

### Auto-Registration: The Opportunity

Rather than requiring the user to edit JSON files manually, the VS Code extension could offer:

**Command: "Agent Kanban: Configure MCP for [Harness]"**

This command would:
1. Detect which harnesses are present (Claude Code's settings file, Cursor config, etc.)
2. Write or patch the appropriate config file
3. Show a success notification: "agentkanban MCP server added to Claude Code config"

Or a single **"Agent Kanban: Configure All Detected Agent Harnesses"** command scans for all known config locations and patches them in one step. This could also run on first activation if Agent Kanban detects a supported harness in the environment.

---

### MCP Server Implementation Shape

The server itself would be a small Node.js process (TypeScript, shipped as `agentkanban-mcp` on npm). It reuses the existing `TaskStore` and `BoardConfigStore` logic:

```
agentkanban-mcp/
  index.ts          # MCP stdio transport entry point
  tools/
    listTasks.ts    # Read board state, return tasks[]
    getTask.ts      # Read full task file content
    selectTask.ts   # Return task + INSTRUCTION.md content (agent bootstrap)
    createTask.ts   # Create new task file
    appendToTask.ts # Append [user] or [agent] turn to task conversation
    moveTask.ts     # Update task lane in frontmatter
```

The `cwd` of the server process is the workspace root, so all `.agentkanban/` paths resolve correctly. No network, no auth — purely local file I/O over stdio.

---

### Summary

| Aspect | Detail |
|--------|--------|
| User experience | Transparent — user talks to agent naturally; MCP calls happen invisibly |
| Registration | One JSON config file per harness; same `command`/`args` pattern everywhere |
| Auto-registration | VS Code extension can patch harness configs via a command |
| Server shape | Small Node.js stdio process reusing existing TaskStore/BoardConfigStore |
| Distribution | `npx agentkanban-mcp` (npm) or bundled in the VS Code extension |
